## 0. 产品定稿

### 0.1 一句话目标

在浏览器中打开即可玩的 **横屏塔防射击游戏**（玩法致敬植物大战僵尸），通过 React 做完整游戏外壳，Phaser 做战斗核心；关卡、植物、僵尸以配置驱动，持续加内容不推倒重来。

### 0.2 明确不做

| 不做 | 原因 |
|------|------|
| 登录 / 注册 / 账号体系 | 你已确认不需要 |
| 服务端、排行榜联网 | 首期无后端，降低复杂度 |
| 竖屏布局 | 只做横屏 16:9 |
| 内购 / 商城支付 | 非目标 |
| 像素级抄袭官方资源 | 用原创命名 + AI 仿风格立绘 |

### 0.3 明确要做（玩法丰富）

- 多植物、多僵尸、多关卡（首发就要「像一款游戏」而不只是 Demo）
- 阳光经济、种植、射击、啃食、波次、大波、胜负
- 植物差异化：产出 / 输出 / 肉盾 / 控制 / 爆炸 / 陷阱 / 辅助
- 僵尸差异化：血量装甲、加速、特殊行为
- 选卡出战、图鉴、本地进度、设置（音量/画质）
- 后期可继续加关卡包、成就、小推车、夜间关等（架构预留）

### 0.4 目标用户与体验

- 桌面浏览器为主（Chrome / Edge / Safari）
- 打开网页 → 点开始 → 选关 → 横屏战场可玩
- 无账号：进度存在浏览器 `localStorage`

## 1. 技术选型（React 配套）

### 1.1 核心栈

| 层级 | 选型 | 用途 |
|------|------|------|
| 构建 | **Vite** | 开发/打包快 |
| 语言 | **TypeScript** | 植物/僵尸/关卡配置有类型，少踩坑 |
| UI 框架 | **React 18** | 页面壳、菜单、弹窗、卡片栏 |
| 路由 | **React Router v6** | 首页 / 选关 / 游戏 / 图鉴 / 设置 |
| 状态 | **Zustand** | 进度、设置、当前局临时状态（比 Redux 轻） |
| 战斗引擎 | **Phaser 3** | 网格、精灵、碰撞、波次、动画 |
| 样式 | **Tailwind CSS** | 快速做横屏 UI |
| 组件库 | **shadcn/ui**（Radix + Tailwind） | 按钮、对话框、进度条、开关、Tabs |
| 图标 | **lucide-react** | UI 图标 |
| 动画（UI） | **framer-motion**（可选） | 菜单过渡、弹窗 |
| 音效（二期） | **howler** | BGM / SFX |
| 工具 | **clsx** + **tailwind-merge** | class 合并 |
| 校验（可选） | **zod** | 校验 levels/plants JSON |

### 1.2 为什么是这套「React 配套」

- **shadcn/ui**：不是笨重组件库，代码在项目里可改，适合游戏菜单定制皮肤。
- **Zustand**：和 Phaser 事件同步简单（`setSun` / `setPaused`），无 Provider 地狱。
- **Phaser 3**：战斗别用 DOM 硬写，可玩性和特效上限更高。
- **Tailwind**：横屏顶栏、卡片栏、弹层布局快。

### 1.3 运行与部署

- 开发：`pnpm dev`（或 npm）本地静态
- 产物：纯前端静态站点，可丢 GitHub Pages / 任意静态托管
- **无后端**

### 1.4 生图（美术资源）

| 项 | 值 |
|----|-----|
| API Base | `http://localhost:8317/v1` |
| Model | `gpt-image-2` |
| 调用方式 | 项目内 `scripts/gen-assets.mjs`，**可并发** |
| 密钥 | 环境变量 `IMAGE_API_KEY` 等，**禁止提交仓库** |
| 策略 | 需要图时自动生成；**无背景/透明底**便于抠图使用 |

#### 1.4.1 生图强制规范（抠图友好）

1. **角色/道具/特效/UI 图标**：必须 **透明背景（alpha）** 或纯色可抠底（优先透明 PNG）。  
2. 提示词必须包含类似：  
   `isolated game sprite, full body or clear subject, **transparent background**, no ground shadow plate, no scenery, centered, clean edges, PNG alpha`  
3. **仅背景图**（草地、菜单背景）允许不透明全幅。  
4. 缺图：开发中发现 `public/assets/**` 缺失 → 立即用脚本并发生成，不阻塞关卡逻辑（可先占位再换）。  
5. 风格锚点（全项目统一）：  
   `Plants vs Zombies inspired original fan art, cute cartoon, clean thick outlines, bright colors, consistent proportions, game asset`

## 3. 玩法设计（丰富版）

### 3.1 基础规则

1. 战场固定 **5 行 × 9 列**。
2. 阳光用于种植；自然掉落 + 向日葵生产 +（可选）关卡初始赠送。
3. 植物只能种在空地；部分植物有特殊格子规则（如睡莲——水关后期）。
4. 僵尸从右侧进入本行，向左走；碰到植物则停止啃食。
5. 僵尸到达最左（房子）→ 触发破防；有小推车则可救一次（后期节点）。
6. 清完所有波次且场上无僵尸 → 胜利。
7. 出战前 **选卡**（卡槽有限，如 6–8 格），增加构筑深度。

### 3.4 关卡规划（首发 12 关，分两章）

#### 第一章：白天草地（1-1 ~ 1-8）

| 关卡 | 名称 | 教学目标 | 强制/推荐植物 | 僵尸重点 |
|------|------|----------|---------------|----------|
| 1-1 | 初见阳光 | 产阳光+射击 | 阳光花、射手豆 | 普通 |
| 1-2 | 筑起防线 | 肉盾 | +硬壳果 | 普通、路障 |
| 1-3 | 一大波 | 波次节奏 | 同上 | +旗手大波 |
| 1-4 | 寒霜将至 | 减速 | +冰霜豆 | 路障群 |
| 1-5 | 爆炸艺术 | 范围清场 | +爆裂果 | 铁桶少量 |
| 1-6 | 脚下留意 | 陷阱 | +地刺薯 | 混合 |
| 1-7 | 火力升级 | 双发+火炬 | +双射豆、火炬木 | 门板、读报 |
| 1-8 | 草地终章 | 综合选卡 | 开放选卡 8 槽 | 橄榄+铁桶+大波 |

#### 第二章：进阶（2-1 ~ 2-4，机制加深）

| 关卡 | 名称 | 新机制 | 说明 |
|------|------|--------|------|
| 2-1 | 刺与盾 | 地刺、高坚果 | 地面伤害教学 |
| 2-2 | 烈焰通道 | 火爆椒 | 整行清场决策 |
| 2-3 | 夜幕将至（简化夜间） | 初始无阳光/少自然阳光 | 强制经济运营 |
| 2-4 | Boss 降临 | 巨人 | 综合考验 + 胜利用作第一章完结 |

每关支持配置：

- `initialSun` / `naturalSun` 开关  
- `cardSlots`  
- `presetCards` 或 `freePick: true`  
- `waves[]`：时间、行、僵尸类型、是否大波  
- `winCondition`：默认清波；可扩展生存时间  
- `starRules`：三星条件（如不丢车、阳光剩余、限时）

### 3.5 经济与手感数值（初稿，实装后调）

| 项 | 建议初值 |
|----|----------|
| 开局阳光（教学关） | 150–200 |
| 自然阳光间隔 | 7–10s |
| 阳光花产额/间隔 | 25 / 24s |
| 豌豆伤害 | 20 |
| 普通僵尸血量 | 200 |
| 路障额外 | +200～370 |
| 铁桶额外 | +800+ |
| 僵尸行走（普通） | 约 1 格 / 若干秒，按手感调 |
| 啃食伤害 | 每秒对植物固定扣血 |

所有数值进配置表，**禁止写死在多个文件**。

### 3.6 操作

- 鼠标：点卡片 → 点格子种植；点阳光收集  
- 快捷键（增强可玩性）：  
  - `1–8` 选卡  
  - `Esc` 暂停  
  - `Space`（可选）收集全场阳光  
- 不强制做触屏拖放；横屏点击即可（Pointer 事件兼容平板横屏）

### 3.7 本地进度（无登录）

```ts
interface ProgressSave {
  version: 1
  unlockedLevelId: string
  stars: Record<string, 0 | 1 | 2 | 3>
  seenCodex: { plants: string[]; zombies: string[] }
  settings: { bgm: number; sfx: number; reduceParticles: boolean }
}
```

- Key：`pvz-web-progress-v1`  
- 设置页可「重置进度」

### 4.1 React ↔ Phaser 通信约定

| 方向 | 事件 | 载荷 |
|------|------|------|
| React→Phaser | `card:select` | `{ plantId }` |
| React→Phaser | `card:deselect` | — |
| React→Phaser | `game:pause` / `game:resume` | — |
| React→Phaser | `game:speed` | `{ rate: 1 \| 1.5 \| 2 }` |
| React→Phaser | `sun:collectAll` | — |
| Phaser→React | `sun:changed` | `{ value }` |
| Phaser→React | `wave:changed` | `{ current, total, huge? }` |
| Phaser→React | `toast` | `{ message, type }` |
| Phaser→React | `game:win` | `{ stars, stats }` |
| Phaser→React | `game:lose` | `{ reason }` |
| Phaser→React | `codex:unlock` | `{ kind, id }` |

`GamePage`：`useEffect` 创建游戏，清理时 `game.destroy(true)`。

## 13. 开发工作流（PR + TDD）

每个功能切片严格走下面闭环（**一个 PR 只做映射表中的节点范围**）：

```text
① Planner 明确本 PR 范围/验收/测试清单
        ↓
② TDD：先写失败测试 → 实现 → 测试变绿
        ↓
③ 推送分支 → 开 GitHub PR
        ↓
④ 并行子 Agent 审查（最多 8 个）
   · planner 对照 PLAN 查范围漂移
   · implementer 自检（可读性/结构）
   · verifier 跑测 + build + 关键路径
   · reviewer 常规 Code Review
   · adversarial 对抗性审查（边界/安全/泄漏/回归）
   ·（可选）game-design 玩法一致性 / a11y / perf
        ↓
⑤ 修复 🔴必须 / 🟡应该 问题 → 复验
        ↓
⑥ 合并 PR → 更新本 PLAN 节点打钩 → 进入下一 PR
```

### 13.1 分支与提交

- 分支：`feature/pr-XXX-<short-name>`（例：`feature/pr-001-scaffold`）
- Commit：Conventional Commits，英文（`feat:` / `test:` / `fix:` / `docs:`）
- 合并：优先 **Squash and merge**
- 合并后删除 feature 分支

### 13.2 TDD 约定

| 层 | 工具 | 测什么 |
|----|------|--------|
| 单元 | Vitest | 数据配置、store、纯函数、伤害/波次逻辑 |
| 组件 | Vitest + Testing Library | 页面壳、选卡、弹窗、路由入口 |
| 游戏逻辑 | Vitest（尽量无 Phaser 头） | Grid/Economy/Wave/Combat 纯逻辑优先可测 |
| 冒烟 | `pnpm build` +（后期）e2e 可选 | 打包不过不得合并 |

规则：

1. **先测后码**：本 PR 新增行为必须先有失败测试。  
2. 纯逻辑与 Phaser 解耦，避免「只能手动点才知道坏了」。  
3. PR 描述必须写：**测试命令** + **结果**。

### 13.3 子 Agent 并行审查（合并前必做）

| Agent | 职责 | 产出 |
|-------|------|------|
| planner | 是否超出 PR 范围、是否完成验收 | 范围报告 |
| implementer | 结构、命名、重复、是否可扩展 | 自检报告 |
| verifier | `pnpm test` / `pnpm build` / 关键脚本 | 通过/失败日志 |
| reviewer | 缺陷、API 误用、类型安全 | Issue 列表 |
| adversarial | 恶意输入、存档损坏、内存泄漏、竞态、加速/暂停边界 | 对抗 Issue |
| （可选）asset | 资源路径、透明底规范、裂图 | 资源报告 |

严重度处理：

- 🔴 bug / 安全 / 测不过 → **必须修完再合**  
- 🟡 设计债 / 可维护性 → 本 PR 能修则修，否则开 follow-up 记入 PLAN  
- 🟢 nit → 不阻断

### 13.4 生图在流程中的位置

- 本 PR 若引入新实体/UI 且需要正式图 → 在实现阶段调用 `scripts/gen-assets.mjs`（透明底）。  
- 生图失败可用色块占位，但须在 PR 描述标注 `assets:placeholder`，并在 N21 清债。

### 13.5 PLAN 打钩规则

- 节点完成且 **对应 PR 已合并 main** → 状态改为 ✅
